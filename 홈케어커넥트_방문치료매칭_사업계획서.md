## 1. 사업 개요

### 1.1 한 줄 요약

> **"AI 기반 위치 매칭으로 방문치료 수요자와 의료기관을 연결하고, 방문치료 운영 전반(스케줄링, 기록, AI 리포트, 레드플래그 감지)을 SaaS로 제공하는 방문치료 통합 플랫폼"**

### 1.2 사업 배경

2026년, 대한민국은 65세 이상 인구가 전체의 20.3%를 넘어 **초고령사회**에 공식 진입했다. 이는 단순한 통계 변화가 아니라 의료 시스템 전반의 패러다임 전환을 요구하는 구조적 변화이다.

```
한국 고령화 추이:
  2022년: 65세 이상 17.4% (고령사회)
  2025년: 65세 이상 20.3% (초고령사회 진입)
  2036년: 65세 이상 30.9% (초초고령사회)
  2050년: 65세 이상 40%+ (OECD 최고 수준)

장기요양 수급자 증가:
  2022년: 101.9만 명
  2023년: 109.8만 명
  2024년: 116.5만 명
  2026년: 130만 명+ (추정)
```

정부도 이에 대응하여 2026년 **「의료·요양 등 지역 돌봄의 통합지원에 관한 법률」** 시행을 앞두고 있으며, 재택의료센터를 전국 250곳까지 확대하는 등 **"병원 중심에서 지역사회·가정 중심으로"** 의료 전달 체계를 전환하고 있다.

그러나 급증하는 방문치료 수요에 비해 **수요자와 의료기관을 효율적으로 연결하는 디지털 인프라**는 사실상 전무한 상태이다.

## 2. 문제 정의

### 2.1 환자/보호자의 문제

```
"엄마가 뇌졸중으로 퇴원하셨는데,
 어디서 방문간호를 받을 수 있는지 모르겠어요..."

현실:
  ① 방문치료 기관 정보가 흩어져 있음 (건보공단, 지자체, 입소문)
  ② 거리·서비스·비용·평판을 한눈에 비교할 수 없음
  ③ 기관에 전화해서 일일이 문의해야 함
  ④ 서비스 시작 후 기록 확인이 어려움 (간호사가 뭘 했는지 모름)
  ⑤ 이상 징후 발생 시 알 방법이 없음 (보호자는 멀리 있음)

결과:
  → 좋은 기관을 놓치고 가까운 곳에 대충 의뢰
  → 간병 부담으로 가족 갈등, 직장 포기
  → 이상 징후 방치 → 응급실 이송 → 의료비 폭증
```

### 2.2 의료기관의 문제

```
방문치료 기관 운영 현실:

  ① 환자 유치: 주로 입소문, 지역 네트워크에 의존
     → 디지털 채널 부재 → 성장 한계

  ② 운영 관리: 엑셀 + 카톡 + 종이 기록
     → 간호사 스케줄링 수작업 → 비효율
     → 방문 기록 취합·정리에 과도한 시간 소요

  ③ 의사 방문 준비: 간호사 기록을 일일이 취합
     → AI 리포트 없이 의사가 직접 차트 검토
     → 월 1회 방문의 가치가 떨어짐

  ④ 품질 관리: 이상 징후 감지 시스템 부재
     → 간호사 개인 역량에 의존
     → 위험 상황 사후 인지

  ⑤ 건보 청구: 수기 서류 작성, 누락·오류 빈번
     → 청구 지연 → 현금 흐름 악화
```

### 2.3 구조적 문제: 수요 폭증 vs 인프라 부재

```
수요 측:
  ┌─────────────────────────────────────┐
  │ 65세 이상 인구: 1,050만 명 (2025)   │
  │ 장기요양 수급자: 130만 명+ (2026)    │
  │ 재가급여 이용자: 전체 수급자의 72%   │
  │ → 연간 10~15% 증가 추세              │
  └─────────────────────────────────────┘

공급 측:
  ┌─────────────────────────────────────┐
  │ 재가장기요양기관: 18,745개소 (2024)  │
  │ 재택의료센터: 250곳 (2026 목표)      │
  │ 방문간호사: 만성 부족                │
  │ → 기관별 편차 극심, 정보 비대칭      │
  └─────────────────────────────────────┘

매칭 인프라:
  ┌─────────────────────────────────────┐
  │           사실상 전무                │
  │  전화 문의 / 입소문 / 건보공단 검색  │
  │  → 이 격차를 홈케어커넥트가 채운다   │
  └─────────────────────────────────────┘
```

## 3. 시장 분석

### 3.1 시장 규모

| 구분 | 규모 | 성장률 | 비고 |
|------|------|--------|------|
| 장기요양보험 총급여비 | 약 12.5조원 (2025) | 연 12~15% | 재가급여 비중 확대 추세 |
| 재가급여 시장 | 약 7.2조원 (2025) | 연 14~18% | 전체 급여의 57%+, 비중 증가 |
| 재가장기요양기관 수 | 18,745개소 (2024) | 연 8~10% | 2008년 대비 715% 증가 |
| 국내 디지털 헬스케어 | 약 2.5조원 (2025) | 연 18% | 원격진료·AI 헬스 포함 |
| 국내 헬스케어 SaaS | 약 4,000억원 (2025) | 연 22% | 병원 EMR, 클라우드 전환 |

### 3.2 TAM / SAM / SOM

```
TAM (전체 시장):
  재가급여 시장 7.2조원 + 디지털 헬스케어 2.5조원
  = 약 9.7조원

SAM (유효 시장):
  재가장기요양기관 18,745곳 × 평균 SaaS 연 매출 기대
  + 환자/보호자 매칭 트래픽
  = 약 3,000억원

SOM (초기 목표 시장):
  수도권 + 광역시 기관 3,000곳 × SaaS 연 600만원
  + 부가서비스
  = 약 200~300억원 (3년 내)
```

### 3.3 시장 성장 동인

```
① 인구 구조적 동인
   65세 이상 인구 매년 40~50만 명 순증
   2036년 30.9% → 수요 지속 확대

② 정책적 동인
   ·2026년 통합돌봄법 시행
   ·재택의료센터 250곳 확대
   ·재가급여 한도액 인상 (등급별 월 2~25만원 증가)
   ·방문진료 시범사업 병원급으로 확대

③ 기술적 동인
   ·AI/LLM 기술 성숙 → 의료 리포트 자동 생성 가능
   ·PWA 기술 → 앱스토어 없이 모바일 서비스 제공
   ·Supabase/Vercel 등 → 소규모 팀도 엔터프라이즈급 인프라

④ 사회적 동인
   ·핵가족화 → 원거리 보호자 증가
   ·맞벌이 가구 → 간병 시간 부족
   ·"Aging in Place" 트렌드 → 시설 대신 가정에서
```

## 4. 솔루션

### 4.1 핵심 개념: AI 매칭 + 운영 SaaS

```
┌─────────────────────────────────────────────────────────┐
│                 홈케어커넥트 솔루션 구조                   │
│                                                         │
│  STEP 1: AI 매칭 (환자 → 기관)                          │
│  ─────────────────────────────────                      │
│  환자 상태 + 위치 + 서비스 유형 입력                      │
│          ↓                                              │
│  AI가 거리·평판·가용인력·서비스 매칭 점수 산출            │
│          ↓                                              │
│  상위 5개 기관 추천 (거리, 점수, 이유 표시)              │
│          ↓                                              │
│  보호자 선택 → 기관에 요청 전달                          │
│  (미응답 48시간 → 자동 2순위 전달)                       │
│                                                         │
│  STEP 2: 초기 평가 → 서비스 계획서                       │
│  ─────────────────────────────────                      │
│  기관 수락 → 간호사 초기 방문 평가                       │
│          ↓                                              │
│  거주 환경·건강 상태 확인                                 │
│          ↓                                              │
│  서비스 계획서 작성 (방문 빈도, 관리 항목, 담당자)        │
│          ↓                                              │
│  보호자 전자 동의 → 서비스 시작                          │
│                                                         │
│  STEP 3: 운영 SaaS (지속 관리)                           │
│  ─────────────────────────────────                      │
│  간호사: GPS 체크인 → 바이탈·기록 입력 → 체크아웃        │
│  AI: 바이탈 추세 분석 → 레드플래그 감지 → 알림 발송      │
│  AI: 월 1회 의사 방문 전 경과 리포트 자동 생성            │
│  보호자: 방문 완료 알림 + 실시간 기록 열람                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 AI 핵심 기능 5가지

**① AI 매칭 엔진**
```
입력:
  환자 위치 (카카오 로컬 API → 좌표 변환)
  필요 서비스: [방문간호, 방문진료, 재활치료, 방문목욕, 방문요양]
  환자 상태: 와상/휠체어/보조기구/자가보행
  선호 시간대: 오전/오후

AI 매칭 알고리즘:
  ① 거리 점수 (PostGIS): 반경 10km 내 기관 필터링
  ② 서비스 적합성: 필요 서비스 제공 여부
  ③ 가용 인력: 간호사 1인당 담당 환자 수 여유
  ④ 평판 점수: 평점, 시간 엄수율, 간호사 근속률
  ⑤ 응답 속도: 평균 요청 응답 시간

출력:
  상위 5개 기관 추천 (점수, 거리, 이동시간, 추천 이유)
```

**② 레드플래그 감지 (AI 이상 징후 경보)**
```
방문 기록 체크아웃 시 자동 트리거:

  Yellow (주의) → 담당 간호사에게만
    · 바이탈 3회 연속 상승 추세
    · 복약 2회 연속 누락
    · 식사량 감소 시작

  Orange (경고) → 간호사 + 의사
    · 바이탈 위험 수치 접근
    · 인지 상태 변화 감지
    · 체중 급격 변화 (2주 내 3kg±)

  Red (긴급) → 간호사 + 의사 + 보호자 + 기관장
    · 수축기 혈압 180+ 또는 90 이하
    · 낙상 발생
    · 의식 변화
    · 상처 급격 악화

핵심 원칙: AI는 "감지"까지만 → 진단/처방 절대 불가
"감지 → 알림 → 의료진 판단" 구조 엄수
```

**③ AI 경과 리포트 (의사 월 1회 방문용)**
```
의사 방문 2~3일 전 자동 생성:

  [리포트 내용]
  ① 환자 기본 정보 요약
  ② 바이탈 추세 그래프 (혈압, 혈당, 체중, 체온)
  ③ 주요 이벤트 (낙상, 식사량 변화, 상처 변화)
  ④ 간호 수행 내역 (드레싱 8회, 혈압 측정 12회 등)
  ⑤ 복약 순응도 (약물별 복용률)
  ⑥ 레드플래그 이력 (경고 내역 + 조치 결과)
  ⑦ AI 종합 요약 (한 문단 요약)

의사: 리포트 확인 후 소견 입력
      → AI가 보호자용 쉬운 말로 변환
      → 의사 확인 후 보호자에게 전달
```

**④ 동선 최적화**
```
간호사 1일 방문 스케줄:

  기존: 간호사가 직접 순서 결정 → 비효율 이동

  홈케어커넥트:
    카카오 모빌리티 API + 실시간 교통 정보
    → 최적 방문 순서 자동 산출
    → 이동 시간 20~30% 절감
    → 하루 1~2명 추가 방문 가능
```

## 6. 핵심 기술

### 6.1 기술 스택

```
Frontend:       Next.js 14+ (App Router) / TypeScript / Tailwind CSS / shadcn/ui
State:          Zustand (클라이언트 상태) / TanStack Query (서버 상태)
Backend:        Supabase (PostgreSQL 15 + Auth + Realtime + Storage + Edge Functions)
AI Engine:      Gemini 2.5 Flash (RAG 챗봇, 리포트 생성, 레드플래그 감지, 음성→텍스트)
Embedding:      gemini-embedding-001 (1536 dim) + Supabase pgvector (HNSW 인덱스)
Search:         BM25 하이브리드 검색 (RAG 키워드 + 벡터 유사도)
Map/Location:   카카오맵 JavaScript API v3 (지도 표시, 마커, 길찾기)
Geocoding:      카카오 로컬 API (주소 → 좌표 변환)
Route:          카카오 모빌리티 API (경로 최적화, 실시간 교통)
Push:           Web Push API (PWA) + 카카오 알림톡 API
Auth:           Supabase Auth (카카오/네이버 OAuth, 휴대폰 본인인증)
Payment:        토스페이먼츠 정기결제 API (SaaS 구독료만)
Storage:        Supabase Storage (상처 사진, 서류, 프로필 이미지)
Hosting:        Vercel (프론트엔드) + Supabase Cloud (백엔드)
Monorepo:       Turborepo + pnpm
```

### 6.2 기술 선택 근거

```
왜 Supabase인가?
  ① PostgreSQL + PostGIS → 위치 기반 거리 계산 네이티브
  ② pgvector → RAG 임베딩 저장/검색 별도 DB 불필요
  ③ Realtime → 보호자 알림, 간호사 위치 실시간 전송
  ④ Edge Functions → 서버리스 AI 엔진 배포
  ⑤ RLS (Row Level Security) → 환자 데이터 접근 제어
  ⑥ 스타트업 초기에 DevOps 부담 최소화

왜 PWA인가?
  ① 앱스토어 심사 없이 즉시 배포 (의료 앱 심사 3~6개월)
  ② 간호사 앱 오프라인 모드 지원 (Service Worker)
  ③ 설치 없이 URL 공유로 보호자 온보딩
  ④ iOS + Android 동시 지원 (개발비 절감)

왜 Gemini 2.5 Flash인가?
  ① 비용 효율성 (GPT-4 대비 1/10 수준)
  ② 한국어 성능 우수
  ③ 긴 컨텍스트 (1M 토큰) → 한 달 치 방문 기록 분석
  ④ 음성→텍스트 + 텍스트 분석 멀티모달
```

| 간병인 매칭 앱 | 케어닥, 위케어 등 | 간병인 1:1 매칭, 기관 운영 도구 없음 | 기관 매칭 + SaaS 운영 도구 통합 |
| 병원 EMR | 여러 EMR 벤더 | 입원 중심, 방문치료 특화 아님 | 방문치료 전 과정 특화 (체크인~리포트) |
| 요양원 매칭 | 케어라인 등 | 시설 매칭, 재가서비스 미지원 | 재가 방문치료 전문 |
| 지자체 돌봄 시스템 | 지자체별 개별 | 분절적, 의료기관 연결 약함 | 전국 통합 플랫폼 |

### 9.2 핵심 경쟁 우위 (해자, Moat)

```
① 네트워크 효과 (양면 시장)
   기관이 많아지면 → 환자 유입 증가 → 기관 더 가입
   "방문치료 = 홈케어커넥트" 인식 선점

② 데이터 해자
   방문 기록 축적 → AI 레드플래그 정확도 상승
   기관 평판 데이터 → 매칭 품질 향상
   → 후발주자가 따라올 수 없는 데이터 자산

③ SaaS 전환 비용 (Lock-in)
   환자 기록, 스케줄, 리포트가 플랫폼에 축적
   타 플랫폼 전환 시 데이터 이전 부담
   간호사 재교육 비용

④ 규제 이해도
   의료법 제27조 (매칭 수수료 불가) 대응 설계
   의료광고 심사 프로세스 내재화
   개인정보보호법 준수 아키텍처
```

## 11. 법률 및 규제 대응

### 11.1 핵심 법률 리스크 및 대응

| 규제 | 내용 | 위반 시 | 대응 전략 |
|------|------|---------|----------|
| **의료법 제27조** | 환자 소개·알선 대가 수수 금지 | 형사처벌 | 매칭 수수료 절대 불가 → SaaS 구독료 모델 |
| **의료법 제56조** | 의료광고 규정 | 과태료·영업정지 | 광고 콘텐츠 심사 후 게재, [광고] 라벨 필수 |
| **의료법 제17조** | 의료행위 비의료인 금지 | 형사처벌 | AI는 "감지"까지만, 진단/처방 절대 불가 |
| **개인정보보호법** | 건강정보 = 민감정보 | 과징금·형사 | 명시적 동의, 접근 로그 전수 기록, 암호화 |
| **의료법 제21조** | 의료기록 보존 5년 | 과태료 | DB 보존 정책, 백업 체계 |
| **장기요양보험법** | 급여비 부당청구 금지 | 환수·정지 | 플랫폼은 청구 "자료 생성"만, 청구 행위 불가 |

### 11.2 법률 안전 설계 원칙

```
① 수익 모델 분리 원칙
   매칭 기능 = 무료 (환자 소개 대가 없음)
   SaaS 기능 = 유료 (운영 도구 사용료)
   → 매칭과 수익이 직접 연결되지 않는 구조

② AI 한계 명시 원칙
   모든 AI 분석 결과에 면책 고지:
   "이 정보는 참고용이며, 의료적 판단은 담당 의료진에게 문의하세요"
   AI는 "감지 → 알림" 역할만 수행
   "진단", "처방", "치료" 용어 사용 금지

③ 비용 표시 원칙
   진료비/본인부담금은 "참고 정보"로만 표시
   플랫폼이 진료비를 수납하지 않음
   결제 기능은 SaaS 구독료만 대상

④ 데이터 보호 원칙
   건강정보 접근: 명시적 동의 + 접근 로그 전수 기록
   데이터 보존: 의료법상 5년 보존 의무 충족
   데이터 공유: 기관 간 건강정보 공유는 별도 동의
   Supabase RLS (Row Level Security) 적용
```

## 12. 리스크 관리

### 12.1 주요 리스크 및 대응

| 리스크 | 영향도 | 발생 가능성 | 대응 전략 |
|--------|--------|-----------|----------|
| **규제 리스크** (의료법 위반) | 매우 높음 | 중간 | 법무 자문 상시, 수익 모델 사전 검증, AI 면책 고지 |
| **기관 온보딩 실패** (기관이 안 씀) | 높음 | 중간 | 무료 기간 충분 제공, CS 인력 투입, 성공 사례 확보 |
| **간호사 앱 사용 거부** | 높음 | 중간 | UX 극도 단순화, 음성 입력, 교육 지원 |
| **보호자 앱 활성도 부족** | 중간 | 높음 | 카카오 알림톡으로 강제 터치포인트, PWA로 설치 부담 제거 |
| **경쟁자 진입** (대형 플랫폼) | 중간 | 중간 | 데이터 해자 선구축, 기관 Lock-in, 규제 이해도 차별화 |
| **AI 오류** (레드플래그 미감지) | 매우 높음 | 낮음 | AI는 보조 수단 명시, 간호사 판단 우선, False Negative 모니터링 |
| **개인정보 유출** | 매우 높음 | 낮음 | RLS, 암호화, 접근 로그, 정기 보안 점검 |
| **수익 모델 검증 실패** | 높음 | 중간 | 무료→유료 전환율 모니터링, 요금제 유연 조정 |

## 결론: 초고령사회의 필수 인프라

```
┌─────────────────────────────────────────────────────────┐
│                   선순환 비즈니스 구조                    │
│                                                         │
│  환자/보호자가 기관을 찾는다                              │
│      ↓                                                  │
│  AI가 최적 기관을 매칭한다                               │
│      ↓                                                  │
│  기관이 SaaS로 방문치료를 운영한다                        │
│      ↓                          ↓                       │
│  [방문 기록 축적]            [보호자 알림·리포트]         │
│  → AI 레드플래그 정확도 ↑    → 보호자 만족도 ↑          │
│  → AI 리포트 품질 ↑          → 입소문 → 환자 유입 ↑     │
│      ↓                          ↓                       │
│  기관 운영 효율 ↑            기관 평판 ↑                 │
│      ↓                          ↓                       │
│  더 많은 기관이 가입 ←──── 더 많은 환자가 유입           │
│      ↓                                                  │
│  데이터 축적 → AI 고도화 → 서비스 품질 ↑                │
│      ↓                                                  │
│  "방문치료 = 홈케어커넥트" 시장 표준화                   │
│                                                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│                                                         │
│  2026년 초고령사회 진입                                  │
│  2036년 노인 인구 30%                                    │
│  → 방문치료 수요는 구조적으로 확대될 수밖에 없다         │
│  → 이 시장의 디지털 인프라를 선점하는 것이 핵심          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

홈케어커넥트는 단순한 매칭 앱이 아니다. **초고령사회에서 "병원 중심"이 아닌 "가정 중심" 의료로 전환되는 거대한 흐름의 디지털 인프라**를 구축하는 사업이다.

정부의 통합돌봄법 시행, 재택의료센터 확대, 재가급여 한도 인상이라는 **정책적 순풍** 속에서, 18,745개 재가기관과 130만 수급자를 연결하는 **양면 시장 플랫폼**으로서의 위치를 선점하겠다.

루미브리즈는 AI 헬스케어 플랫폼 개발 역량과 의료 도메인 이해를 바탕으로, **대한민국 방문치료의 디지털 표준**을 만들어 나가겠다.
